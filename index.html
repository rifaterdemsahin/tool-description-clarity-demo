🔍 Step 0: The Problem — What's Happening?

The Customer Support Resolution Agent frequently calls get_customer when users ask about order status ("Where is my order ORD-9012?"). The correct tool is lookup_order. Users receive customer profile data when they asked for order tracking — a fundamental tool-selection failure.

💡 WHY: The semantic description string is the PRIMARY feature an LLM uses to match user intent to tool schema. When descriptions are vague, the LLM cannot distinguish which tool handles which entity domain.

❌ Step 1: The Naive Approach — Why It Fails

The naive tool registry has vague, overlapping descriptions: "Gets a customer record" and "Looks up an order". Both use generic verbs ("gets", "looks up") with no distinguishing entity scope. When the LLM encounters the word "order" in a user query, it has no semantic anchor to exclude get_customer — ~60% of order-status queries route to the wrong tool.

⚠️ HOW it fails: The LLM's embedding of the query text has high cosine similarity to both tool descriptions. Without explicit entity-boundary signals, the selection is effectively random — and statistically biased toward whichever tool has broader semantic coverage.

🔤 Token waste: Each wrong call triggers a retry loop → 800-1200 extra input tokens per query consumed in correction prompts.

✅ Step 2: The Resilient Architecture — How It Works

The resilient tool registry rewrites descriptions with explicit entity boundaries and exclusion clauses:

❌ "Gets a customer record from the system."

No entity scope. Matches any query.

✅ "Retrieves a CUSTOMER profile by id. Use for account details — NOT for order status."

Explicit scope. LLM matches correctly.

🛡️ HOW the three pillars work: (1) Context:Fork isolates entity domains with NOT-clauses, (2) Clear semantic labels use distinct entity names (CUSTOMER vs ORDER), (3) Argument hints guide parameter-level correct invocation. Together they form a declarative defense-in-depth for tool selection.

📋 Step 4: Domain Model — Tool Schemas & Entities

The domain model defines three core entity types (Customer, Order, ToolDefinition) and four tools. The description field on each tool is the architectural seam — it's the only component the LLM reads to decide which tool to invoke. Getting it right is the single highest-leverage fix.

💡 Step 6: Key Takeaway — Fix Descriptions FIRST

Before building complex classifier wrappers, before dumping hundreds of few-shot examples into prompts, before adding routing middleware — review the tool descriptions. The semantic description string is the primary feature the LLM uses to map user intent to tool execution schemas. A clear, well-scoped description eliminates the need for all downstream complexity.
